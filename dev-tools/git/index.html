第 1 章 Git 是什麼 & 你需要的基本觀念
🧠 步驟 1：用 3 句話理解 Git

Git 是一套版本控制系統，每次 commit 就像替專案拍一張快照。你可以隨時回到任意一張快照，也可以讓多人同時在不同快照上工作再合併。

📦 步驟 2：認識 Repository、Working Tree、Staging Area

Repository（repo）：存放所有歷史版本的資料庫，藏在 .git/ 資料夾裡。

Working Tree：你現在看到、正在編輯的檔案區。Staging Area（Index）：你手動 git add 後，準備放進下一次 commit 的暫存區。

[Working Tree] --git add--> [Staging Area] --git commit--> [Repository]
🧱 步驟 3：認識 Commit（為什麼要小步提交）

每個 commit 都有唯一的 hash（如 a3f8c2d），記錄「誰、何時、改了什麼」。小步提交讓回溯更精確，也讓隊友 code review 更容易聚焦。

🌿 步驟 4：認識 Branch（分支是指標，不是資料夾）

Branch 只是一個指向某個 commit 的「指標」，切換分支不會複製檔案，速度非常快。HEAD 表示你現在「站」在哪個 commit 上。

main  --> commit C
feature --> commit D (從 C 長出來)
🌍 步驟 5：認識 Remote（origin 是什麼）

Remote 是遠端伺服器上的 repo（通常在 GitHub / GitLab）。origin 是 Git 預設給遠端的名稱，就像「遠端的暱稱」。

git remote -v
# origin  https://github.com/yourteam/project.git (fetch)
# origin  https://github.com/yourteam/project.git (push)
💡 本章提示卡
🟢 實用技巧
把 Git 想成「時光機 + 平行宇宙」：commit 是時間點，branch 是平行線，merge 是把兩條線合回一條。
🟡 注意事項
.git/ 資料夾是 Git 的心臟，不要手動刪除或編輯裡面的檔案。
🔴 常見錯誤
以為「branch = 資料夾」，其實切換 branch 後你的 Working Tree 內容會跟著變，這是正常的。
🔵 補充說明
一個本機 repo 可以對應多個 remote（如同時推 GitHub 和 GitLab），不只限於 origin。
第 2 章 安裝與初次設定
✅ 步驟 1：確認已安裝

打開 PowerShell 或終端機，執行以下指令確認 Git 已安裝且版本正常。若未安裝，前往 git-scm.com 下載。

git --version
# 輸出範例：git version 2.44.0.windows.1
👤 步驟 2：設定身分（必做）

每次 commit 都會帶上這個身分資訊，讓隊友知道是誰改的。--global 表示全機器套用。

git config --global user.name "你的名字"
git config --global user.email "[email protected]"
🧾 步驟 3：檢查目前設定

用以下指令確認 name/email 已正確設定，也可看到所有生效的設定項目。

git config --list
git config user.name   # 只看特定項目
🧷 步驟 4：Windows 換行符設定（core.autocrlf）

Windows 預設換行是 CRLF，Linux/macOS 是 LF。跨平台協作容易產生不必要的 diff，建議設定如下：

# Windows 開發者（推薦）
git config --global core.autocrlf true
# 儲存時轉 LF，checkout 時還原 CRLF

# macOS / Linux 開發者
git config --global core.autocrlf input
💡 本章提示卡
🟢 實用技巧
設定 default branch 名稱為 main（新版 Git 慣例）：git config --global init.defaultBranch main
🟡 注意事項
公司電腦若是共用帳號，建議不要用 --global，在專案裡單獨設定即可（去掉 --global）。
🔴 常見錯誤
email 填錯或沒設定，導致 GitHub 上 commit 顯示為「匿名」，貢獻圖不會記錄。
🔵 補充說明
設定檔儲存在 ~/.gitconfig（Windows 為 C:\Users\你的帳號\.gitconfig），可直接用文字編輯器打開修改。
第 3 章 建立或取得專案
🆕 步驟 1：新專案初始化（git init）

在現有資料夾裡建立 Git repo，會產生 .git/ 子資料夾。之後就可以開始 add/commit。

mkdir my-project
cd my-project
git init
# Initialized empty Git repository in .../my-project/.git/
🔗 步驟 2：取得既有專案（git clone）

從遠端下載整個 repo，包含所有歷史。clone 後會自動設好 origin 指向來源。

git clone https://github.com/yourteam/project.git
# 或指定本機資料夾名稱
git clone https://github.com/yourteam/project.git my-folder
📁 步驟 3：確認目前狀態（git status）

這是最常用的指令之一，隨時告訴你現在有哪些未追蹤/已修改/已暫存的檔案。

git status
# On branch main
# nothing to commit, working tree clean
🧭 步驟 4：看遠端指向（git remote -v）

確認這個 repo 連接的遠端網址正確，也可以用來切換遠端（如搬家到新的 GitLab）。

git remote -v
# origin  https://github.com/yourteam/project.git (fetch)
# origin  https://github.com/yourteam/project.git (push)
💡 本章提示卡
🟢 實用技巧
用 SSH clone（[email protected]:...）可以省掉每次 push 輸入帳密，推薦在自己的電腦上設定 SSH Key。
🟡 注意事項
git init 後記得先建 .gitignore，避免把 node_modules、.env 等敏感檔案加進去。
🔴 常見錯誤
在根目錄（如 C:\）不小心執行 git init，導致整個硬碟被 Git 管理。請馬上 rm -rf .git（PowerShell: Remove-Item .git -Recurse）。
🔵 補充說明
git clone --depth 1 只下載最新一筆歷史，適合大型 repo 快速取得最新版本。
第 4 章 第一次提交流程
🔍 步驟 1：看有哪些變更（git status）

養成習慣：每次要 add 之前先看 status，確認是否有不預期的修改。

git status
# Changes not staged for commit:
#   modified:   src/app.js
# Untracked files:
#   config.json
➕ 步驟 2：加入暫存區（git add）

只有放進暫存區的變更才會被下次 commit 包含。用 . 加入全部，或指定單一檔案更精確。

# 加入所有變更
git add .

# 只加入特定檔案
git add src/app.js

# 互動式選擇（進階）
git add -p
📝 步驟 3：建立提交（git commit）

用 -m 直接附上訊息；訊息要能讓三個月後的自己看懂這次改了什麼。

git commit -m "feat: 新增使用者登入功能"

# 如果想要多行訊息，直接 git commit（會開編輯器）
📜 步驟 4：檢視提交歷史（git log）

加上 --oneline --graph --decorate 可以看到簡潔的分支樹狀圖，非常好用。

git log --oneline --graph --decorate
# * a3f8c2d (HEAD -> main) feat: 新增使用者登入功能
# * 9d1b4e7 fix: 修正 API 回傳格式
# * 3c0f118 init: 初始化專案
🧹 步驟 5：.gitignore 的用途與常見規則

在專案根目錄建立 .gitignore 檔案，列出不要追蹤的檔案或資料夾，Git 就不會提示你去 add 它們。

# .gitignore 常見範例
node_modules/
.env
.env.local
dist/
*.log
.DS_Store       # macOS
Thumbs.db       # Windows
💡 本章提示卡
🟢 實用技巧
git commit --amend 可以修改「最後一次」commit 的訊息或補加檔案，但只限還沒 push 出去時用。
🟡 注意事項
git add . 會把當前目錄所有變更都加進去，包括你沒意識到的修改，記得先 git status 確認。
🔴 常見錯誤
把 .env（含 API Key、資料庫密碼）commit 進 repo 推上 GitHub——這是資安事故，請立刻撤銷並重設密鑰。
🔵 補充說明
gitignore.io（toptal.com/developers/gitignore）可以按語言/框架產生 .gitignore 範本，非常方便。
第 5 章 分支基本操作
🌿 步驟 1：列出與建立分支

git branch 列出本地所有分支，加 -a 也顯示遠端分支。switch -c 是新版建立並切換的推薦指令。

# 列出分支（* 號代表目前所在）
git branch
git branch -a   # 含遠端

# 建立新分支（不切換）
git branch feature/login

# 建立並立刻切換（推薦）
git switch -c feature/login
🧭 步驟 2：切換分支（git switch）

切換前確認 Working Tree 是乾淨的（沒有未提交的變更），否則會被擋下或帶著修改跑到新分支。

git switch main          # 切到 main
git switch feature/login # 切到功能分支

# 舊版指令（仍可用）
git checkout main
🔀 步驟 3：合併分支（git merge）

先切到「想接收合併」的分支（通常是 main），再執行 merge。Git 會把另一個分支的 commit 整合進來。

# 把 feature/login 合進 main
git switch main
git merge feature/login

# 合併完畢後可以刪除功能分支
git branch -d feature/login
🏷️ 步驟 4：常用分支命名習慣

統一命名讓所有人一眼看懂這條分支是做什麼的，也方便 CI/CD 設規則。

feature/user-login       # 新功能
bugfix/fix-null-pointer  # 修 Bug
hotfix/critical-crash    # 緊急修復（從 main 開）
release/v1.2.0           # 版本發布準備
chore/update-deps        # 雜項維護
💡 本章提示卡
🟢 實用技巧
git merge --no-ff feature/login 強制產生 merge commit，讓歷史中看得到分支脈絡，團隊環境推薦使用。
🟡 注意事項
刪除分支只是刪指標，commit 本身還在。但超過一段時間後 GC 會清掉，所以請確認合併完才刪。
🔴 常見錯誤
在 main 分支直接開發並 push，沒有用 feature branch，導致多人同時修改衝突嚴重。
🔵 補充說明
git switch 是 Git 2.23 新增的指令，比 checkout 更語意清楚，建議優先使用。
第 6 章 與遠端協作
📡 步驟 1：更新遠端資訊（git fetch）

fetch 只下載遠端的最新資訊，不會自動合進你的 Working Tree，是最安全的同步方式。

git fetch origin
# 也可以 fetch 所有遠端
git fetch --all
⬇️ 步驟 2：取得最新變更（git pull）

pull = fetch + merge，會直接合進當前分支。建議加 --rebase 讓歷史更線性。

git pull
# 或用 rebase 模式（減少 merge commit）
git pull --rebase origin main
⬆️ 步驟 3：推送分支到遠端（git push）

第一次推新分支需要加 -u 設定追蹤，之後就可以直接 git push。

# 第一次推（設定 upstream）
git push -u origin feature/login

# 之後直接推
git push
🧷 步驟 4：upstream 是什麼（為什麼第一次要 -u）

-u（--set-upstream）讓本地分支記住對應的遠端分支，之後 push/pull 就不用再指定名稱。

# 設定完 upstream 後，git push/pull 就知道要對哪裡
# 沒設的話每次都要寫完整：git push origin feature/login
🧾 步驟 5：看分支追蹤狀態（git branch -vv）

確認每個本地分支追蹤了哪個遠端分支，以及本地是否超前或落後幾個 commit。

git branch -vv
# * feature/login a3f8c2d [origin/feature/login: ahead 2] feat: 新增登入
# main            9d1b4e7 [origin/main] fix: 修正 API
💡 本章提示卡
🟢 實用技巧
每天開始工作前先 git fetch 再 git status，掌握目前分支與遠端的差距，再決定要 pull 還是 rebase。
🟡 注意事項
有人 force push 遠端後，你的 git pull 可能會失敗或產生奇怪歷史，這時要先確認狀況再操作。
🔴 常見錯誤
直接 git push origin main 蓋掉別人的提交——push 前請先 pull/fetch 確認沒有新的遠端 commit。
🔵 補充說明
macOS/Linux 設定預設 pull 行為：git config --global pull.rebase true，讓 pull 預設用 rebase 模式。
第 7 章 解衝突
💥 步驟 1：什麼時候會衝突

當兩個分支都修改了「同一檔案的同一個區塊」，Git 無法自動判斷要用哪個版本，就會標記衝突要你手動決定。

# 衝突常發生在：
# 1. git merge feature/xxx 時
# 2. git pull（有人改了同一行）
# 3. git rebase 過程中
🧩 步驟 2：看到衝突標記怎麼讀

<<<<<<< 到 ======= 是你的版本；======= 到 >>>>>>> 是對方的版本，你需要保留其中一份或合寫一份。

<<<<<<< HEAD
const greeting = "Hello, World!";
=======
const greeting = "Hi there!";
>>>>>>> feature/update-greeting
🛠️ 步驟 3：解衝突標準流程

用編輯器（或 VS Code 的 Merge Editor）打開衝突檔案，手動選擇要留的內容，然後 add 再 commit。

# 1. 用 VS Code 打開解衝突（推薦）
code .   # 衝突的檔案會有特殊標示

# 2. 解完後（刪掉 <<<, ===, >>> 標記）
git add src/greeting.js

# 3. 完成 merge commit
git commit -m "merge: 解決 greeting 衝突"
🧪 步驟 4：解完怎麼自檢

解衝突最怕手滑刪掉有用的程式碼，務必跑測試或至少看一次 diff，確認程式仍可執行。

git diff HEAD        # 看目前所有變更
git status           # 確認沒有還沒解的衝突
# 然後跑你的測試：npm test / pytest / dotnet test ...
🧯 步驟 5：放棄合併（git merge --abort）

如果衝突太複雜或現在不想處理，可以直接放棄這次合併，還原到 merge 前的狀態。

git merge --abort
# 也適用於 rebase 中途放棄
git rebase --abort
💡 本章提示卡
🟢 實用技巧
VS Code 的 Merge Editor 有「接受自己 / 接受對方 / 兩者都保留」按鈕，比手動刪標記快很多。
🟡 注意事項
解衝突時記得刪掉所有 <<<<<<<、=======、>>>>>>> 標記，否則程式會直接 syntax error。
🔴 常見錯誤
解完衝突忘記 git add，直接 commit 結果 Git 說「nothing to commit」，因為衝突檔案還在 unmerged 狀態。
🔵 補充說明
減少衝突的最佳策略：分支活太久就會和 main 差太遠，養成習慣每天把 main 的最新 commit 同步進功能分支（git merge main 或 git rebase main）。
第 8 章 回溯與救援
🧼 步驟 1：丟棄工作區變更（git restore）

把某個檔案（或所有檔案）還原成最後一次 commit 的樣子，未提交的修改會消失。

# 還原特定檔案
git restore src/app.js

# 還原所有修改（危險！不可逆）
git restore .

# 把暫存區的檔案退回工作區
git restore --staged src/app.js
↩️ 步驟 2：回到某個提交（git reset 三種模式）

三種模式的差異在於「working tree」和「staging area」要不要跟著重置，選錯會讓你的改動憑空消失。

# --soft：只移動 HEAD，暫存區和工作區不動（最安全）
git reset --soft HEAD~1

# --mixed（預設）：移動 HEAD + 清空暫存區，工作區保留
git reset HEAD~1

# --hard：三者都重置，修改全部消失（高風險）
git reset --hard HEAD~1
🧾 步驟 3：安全反轉（git revert）

不修改歷史，而是新增一個「撤銷某次 commit 效果」的新 commit，已 push 到遠端時的首選做法。

# 反轉最新一次 commit
git revert HEAD

# 反轉特定 commit（需要該 commit 的 hash）
git revert a3f8c2d
🧭 步驟 4：反悔救援（git reflog）

reflog 記錄了你本機 HEAD 的所有移動紀錄，即使 reset --hard 之後，也能用 reflog 找回那個 commit hash 再 checkout 救援。

git reflog
# a3f8c2d HEAD@{0}: reset: moving to HEAD~1
# 9d1b4e7 HEAD@{1}: commit: feat: 新增登入功能  ← 這個要救

# 找到 hash 後 checkout 回來
git switch -c rescue-branch 9d1b4e7
💡 本章提示卡
🟢 實用技巧
記憶口訣：soft = 退回改動到暫存區，mixed = 退回到工作區，hard = 改動直接消失。
🟡 注意事項
git revert 才是已推送 repo 的正確回溯方式，因為它不改歷史；reset 已推送的 commit 會需要 force push。
🔴 常見錯誤
用 git restore . 時沒意識到工作區所有未提交修改都會消失，兩小時的工作付之一炬。
🔵 補充說明
reflog 預設只保留 90 天，超過後才真的消失，所以大部分情況都還救得到。
⚠️ 高風險操作警告：reset --hard 與 force push
  • git reset --hard：Working Tree 所有未提交變更立刻永久消失，無法用 Ctrl+Z 救回，使用前請再三確認。
  • git push --force / push --force-with-lease：會覆蓋遠端歷史，其他人的 pull 可能出現衝突或遺失 commit。請只在個人私有分支上使用，絕對不要 force push main / develop。
  • shared branch 上的 reset：若已有人基於這個 commit 開了新分支，你 reset 會讓他們的 branch 懸空，造成難以追蹤的問題。
第 9 章 提交整理與品質
✍️ 步驟 1：好的 commit message 格式

推薦使用 Conventional Commits 格式：type: 簡短描述，讓 changelog 自動生成更容易。

# 格式：type(scope): 描述（全部小寫，50字以內）
feat: 新增 Google OAuth 登入
fix: 修正手機版導覽列跑版
docs: 更新 README 安裝步驟
chore: 升級 Node.js 到 20.x
refactor: 將 auth 邏輯抽出為 service
test: 新增 login API 的單元測試

# 多行格式（首行 + 空行 + 詳細說明）
feat: 新增購物車結帳流程

支援信用卡、Line Pay 兩種付款方式。
解決 issue #123：金額計算精度問題。
🧽 步驟 2：合併多個提交（squash 的概念與時機）

開發過程中的 WIP commit 太多太雜，合進 main 前可以用 interactive rebase 把它們壓成一個乾淨的 commit。

# 互動式 rebase：整理最近 3 個 commit
git rebase -i HEAD~3

# 編輯器中把想合併的 commit 改成 squash 或 s
# pick a3f8c2d feat: 新增登入
# squash 9d1b4e7 WIP: 修一下
# squash 3c0f118 WIP: 再修一下
🧬 步驟 3：rebase 的用途與風險

rebase 可以把你的分支「搬移」到最新的 main 上，讓歷史看起來更線性、漂亮。但在共用分支上用 rebase 會改寫歷史，造成隊友混亂。

# 把 feature 分支搬到最新 main 上
git switch feature/login
git rebase main

# 解 rebase 衝突的流程
# 1. 解完衝突後
git add .
git rebase --continue
# 或放棄
git rebase --abort
🔍 步驟 4：PR 前自我檢查

送 PR 前花 5 分鐘自我審查，可以減少 review 來回次數，也讓 reviewer 更好審。

# 看這次分支改了哪些東西
git diff main...HEAD

# 看 commit 歷史是否乾淨
git log main..HEAD --oneline

# 確認狀態乾淨
git status
💡 本章提示卡
🟢 實用技巧
commit message 可以用英文或中文，但同一個 repo 請統一。加上 issue 編號（如 fix #42: ...）讓 GitHub 自動關聯。
🟡 注意事項
squash 之後原本的 commit hash 都會改變，所以請在 squash 完再 push，不要先 push 再 squash。
🔴 常見錯誤
在已有多人 pull 的共用分支上執行 rebase，導致大家的 branch 和遠端歷史分叉，要花很多時間修。
🔵 補充說明
可以用 git log --author="你的名字" --oneline -20 快速回顧你最近的提交品質，當作自我 review。
⚠️ rebase 與 force push 使用守則
  • 只在個人功能分支（feature/* / bugfix/*）上使用 rebase，絕對不要 rebase main、develop 等共用分支。
  • rebase 後必須用 git push --force-with-lease（而非 --force），它會先確認遠端沒有新 commit，更安全。
  • 如果不確定有沒有其他人基於你的分支，請先溝通再決定要不要 rebase + force push。
第 10 章 常見情境速查
😵 步驟 1：忘記 add / commit 某個檔案

最後一次 commit 漏加了檔案，用 --amend 補進去，不用新增一個 commit。

# 補加漏掉的檔案
git add config.json
git commit --amend --no-edit
# --no-edit 表示不修改 commit message
🧯 步驟 2：提交到錯的分支

把不該在 main 上的 commit 搬去正確的 feature 分支，再把 main 退回去。

# 情境：不小心在 main commit 了一個 feature
# 步驟 1：把這個 commit 的 hash 記下來（git log --oneline）
# 步驟 2：切到正確分支並 cherry-pick 過來
git switch feature/login
git cherry-pick a3f8c2d

# 步驟 3：把 main 退回（soft 保留改動）
git switch main
git reset --soft HEAD~1
🧵 步驟 3：只想拿某個 commit（git cherry-pick）

不合併整條分支，只把特定的 commit 搬過來，適合 hotfix 要同時套到多個版本分支的場景。

# 把 a3f8c2d 這個 commit 套用到當前分支
git cherry-pick a3f8c2d

# 套用多個
git cherry-pick a3f8c2d 9d1b4e7

# 套用一段範圍（不含起點）
git cherry-pick a3f8c2d..9d1b4e7
🧼 步驟 4：工作做到一半要切換（git stash）

臨時要切分支但不想 commit 半成品，用 stash 把目前改動「藏起來」，切完再恢復。

# 藏起來
git stash push -m "WIP: 登入頁面樣式"

# 切換分支做事...
git switch hotfix/crash-fix

# 切回來，恢復藏的改動
git switch feature/login
git stash pop     # 或 git stash apply stash@{0}

# 查看所有 stash
git stash list
🧪 步驟 5：看差異（git diff 常用參數）

在 commit 前養成看 diff 的習慣，能避免很多「不小心改到的」錯誤。

# 工作區 vs 暫存區
git diff

# 暫存區 vs 最後一次 commit
git diff --staged

# 兩個分支的差異
git diff main feature/login

# 某個 commit 和它的上一個
git diff a3f8c2d^ a3f8c2d

# 只看哪些檔案有差（不看內容）
git diff --name-only main
💡 本章提示卡
🟢 實用技巧
stash 也可以暫存 untracked 的新檔案：git stash push -u -m "WIP"，加 -u 把新檔案也一起藏。
🟡 注意事項
cherry-pick 可能產生衝突，跟 merge 衝突的解法一樣，解完後 git cherry-pick --continue。
🔴 常見錯誤
stash pop 後忘記自己有東西被藏著，之後又 stash 一次導致多個 stash 堆積，最後搞不清楚哪個是哪個。定期 git stash list 清理一下。
🔵 補充說明
git commit --amend 只能修改最後一次 commit，更早的 commit 要用 git rebase -i 才能修改。
第 11 章 團隊流程建議
🧭 步驟 1：建議分支策略（簡化版 Git Flow）

對大多數團隊來說，三層分支結構已經夠用：main（正式環境）、develop（整合測試）、feature/*（開發）。

main          ← 只接受 PR，永遠可以部署
  └─ develop  ← 功能整合、測試環境
       ├─ feature/user-profile
       ├─ feature/payment-flow
       └─ bugfix/login-redirect

# hotfix 從 main 開分支，修完 merge 回 main 和 develop
hotfix/critical-crash
✅ 步驟 2：PR 前檢查清單

送 PR 前自我審查五個項目，可以大幅減少 review 往返次數。

PR 前自我檢查：
□ 程式碼可正常編譯 / 不會 lint error
□ 相關的測試都有通過
□ 程式碼格式化（prettier / eslint）
□ commit 歷史乾淨（沒有 WIP / temp 等亂提交）
□ PR 標題 & 描述清楚說明目的和影響範圍
□ 有截圖（UI 變更時）或測試步驟說明
🔒 步驟 3：保護分支概念（避免直接推 main）

在 GitHub/GitLab 的 Branch Protection Rules 設定保護 main：要求至少 1 人 approve PR、CI 必須通過，才能合併。

GitHub 設定路徑：
Settings → Branches → Branch protection rules → Add rule

推薦設定：
✓ Require a pull request before merging
✓ Require approvals (1 人以上)
✓ Require status checks to pass (CI/CD)
✓ Do not allow bypassing the above settings
🧩 步驟 4：Code Review 的基本禮儀

PR 越小越容易 review，reviewer 花 10 分鐘就能看完比花 1 小時只看懂一半好。給意見時說清楚是 blocking（必改）還是 nit（可選）。

# 好的 PR 原則：
- 每個 PR 只做一件事
- PR description 說明「為什麼」不只是「改了什麼」
- 自己先 review 一遍再發出去

# 好的 review 評論範例：
[nit] 這個變數名可以更語意化
[blocking] 這邊沒有處理 null case，會噴 error
[suggestion] 可以考慮抽出 helper function，但不是必要的
💡 本章提示卡
🟢 實用技巧
用 git log --oneline --graph --all 畫出整個 repo 的分支圖，讓團隊成員在每週 sync 時一眼看清楚目前進度。
🟡 注意事項
分支命名要有一致的 prefix（feature/ bugfix/ hotfix/），方便 CI/CD 依名稱自動觸發不同的部署流程。
🔴 常見錯誤
PR 太大（超過 500 行變更），reviewer 根本無法有效審查，最後只能 approve 了事，失去 review 意義。
🔵 補充說明
Draft PR 功能（GitHub/GitLab 都支援）適合提早分享進度、得到早期 feedback，但還沒準備好被 merge 時使用。
