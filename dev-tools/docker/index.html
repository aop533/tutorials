第 1 章Docker 是什麼？核心概念總覽
😩 步驟 1：傳統部署的痛點

「我電腦跑得起來，你那邊怎麼跑不起來？」這句話相信你一定聽過。原因是每個人的作業系統、語言版本、套件版本都不一樣，光是「把環境弄一致」就能耗掉半天。

Docker 就是用來解決這個問題的——它把「環境」打包進容器，讓你在任何機器上都能跑出一模一樣的結果。

📦 步驟 2：Image 是藍圖，Container 是房子

Image：唯讀的模板，記錄了「跑這個服務需要什麼環境、哪些檔案」。就像蓋房子的設計圖，本身不會執行。

Container：Image 實際跑起來的實例。同一張 Image 可以同時跑出多個 Container，每個都是獨立的。

# Image → Container 的關係
Image（設計圖）  →  docker run  →  Container（實際運行的房子）
                               →  Container（另一棟，互不影響）
🌐 步驟 3：Registry 是倉庫（Docker Hub）

Docker Hub 是官方的公開 Image 倉庫，存放了數十萬個現成 Image（nginx、mysql、node、python…）。你可以直接 pull 下來用，也可以把自己的 Image push 上去分享。

docker pull nginx        # 從 Docker Hub 下載 nginx Image
docker push myapp:1.0   # 推送自己的 Image 到 Registry
🏗️ 步驟 4：Docker 整體架構一眼看懂

Docker 的運作架構由三層組成：你下指令 → Docker Engine 執行 → 容器跑起來。

你的電腦（Host）
└── Docker Engine（守護程序，後台執行）
    ├── Container A（nginx）
    ├── Container B（mysql）
    └── Container C（你的 App）
🆚 步驟 5：Container vs 虛擬機（VM）

VM 是把整個作業系統虛擬化，很重；Container 只隔離程序和環境，共用 Host 的 OS kernel，啟動只要幾秒，資源消耗也小得多。

VM：  Host OS → Hypervisor → Guest OS → App（啟動慢，檔案大）
容器：Host OS → Docker Engine → Container → App（啟動秒級，輕量）
💡 本章提示卡
🟢 實用技巧
記憶口訣：Image = 食譜，Container = 炒出來的菜。同一份食譜可以炒出很多盤，但每盤都是獨立的。
🟡 注意事項
Docker 不是虛擬機！Container 沒有自己的 OS kernel，共用 Host 的，所以比 VM 輕很多，但也因此 Windows Container 和 Linux Container 不能混跑。
🔴 常見錯誤
以為「刪掉 Container 就會失去 Image」——不會！Image 還在，隨時可以再 docker run 跑一個新的 Container。刪掉的只是那個執行中的實例。
🔵 補充說明
Docker 背後的底層技術是 Linux 的 cgroups + namespaces，這也是為什麼 Windows 上要透過 WSL2（Linux kernel）才能跑 Linux Container。

第 3 章Windows：WSL2 + Ubuntu + Docker Desktop 實戰安裝
🪟 步驟 1：啟用 WSL2 功能

用系統管理員身分開啟 PowerShell，執行以下指令。完成後會要求重開機。

# 以系統管理員執行 PowerShell
wsl --install

# 若已有 WSL1，升級為 WSL2
wsl --set-default-version 2

# 重開機後確認
wsl --status
🐧 步驟 2：安裝 Ubuntu 發行版

從 Microsoft Store 搜尋「Ubuntu」安裝（建議 Ubuntu 22.04 LTS），或用指令安裝：

# 查看可用的 Linux 發行版
wsl --list --online

# 安裝 Ubuntu 22.04
wsl --install -d Ubuntu-22.04

# 安裝完成後設定使用者名稱與密碼
# 啟動 Ubuntu
wsl -d Ubuntu-22.04
🐳 步驟 3：下載並安裝 Docker Desktop for Windows

前往 docker.com 下載 Docker Desktop for Windows，安裝過程中確認勾選「Use WSL 2 instead of Hyper-V」。

# 安裝完成後，在 Docker Desktop 設定：
# Settings → Resources → WSL Integration
# → 開啟 "Enable integration with my default WSL distro"
# → 勾選 Ubuntu-22.04
🔗 步驟 4：在 WSL2 Ubuntu 裡測試 Docker

打開 Ubuntu Terminal（不是 PowerShell），測試 Docker 是否正常運作：

# 在 WSL2 Ubuntu 內執行（不是 PowerShell）
docker --version
# Docker version 25.x.x, build ...

docker run hello-world
# 出現 "Hello from Docker!" 表示成功
⚠️ 步驟 5：重要！不要在 Ubuntu 裡再裝一套 Docker

Docker Desktop 已經透過 WSL2 整合提供了 Docker Engine，你在 Ubuntu 裡直接用就好。如果又在 Ubuntu 裡用 apt 安裝 Docker Engine，會出現兩個 daemon 衝突的問題。

# ❌ 不要在 WSL2 Ubuntu 裡執行這個
sudo apt install docker.io   # 會造成衝突

# ✅ 正確做法：Docker Desktop 設定好後直接在 Ubuntu 用
docker ps   # 直接可以用，不需要 sudo
💡 本章提示卡
🟢 實用技巧
安裝 VS Code 並加裝「WSL」擴充套件，之後在 Ubuntu 裡執行 code . 就能直接用 VS Code 開啟 WSL2 裡的專案，體驗幾乎跟原生 Linux 一樣。
🟡 注意事項
Docker Desktop 必須在 Windows 背景執行，WSL2 裡的 docker 指令才能用。如果 Docker Desktop 沒開，會出現 "Cannot connect to the Docker daemon" 錯誤。
🔴 常見錯誤
在 PowerShell 裡執行 docker 指令——雖然可以用，但建議養成習慣全部在 WSL2 Ubuntu 裡操作，路徑和權限問題會少很多。
🔵 補充說明
WSL2 使用完整的 Linux kernel，效能比 WSL1 好很多。如果你的電腦之前裝過 WSL1，記得用 wsl --set-version Ubuntu 2 升級。
第 4 章macOS 與 Linux：原生 Docker 安裝與基本設定
🍎 macOS 步驟 1：下載 Docker Desktop for Mac

前往 docker.com/products/docker-desktop 下載，注意要選對版本：

# 確認你的 Mac 晶片
uname -m
# x86_64  → 下載 Intel 版
# arm64   → 下載 Apple Silicon 版（M1/M2/M3）
🍎 macOS 步驟 2：安裝與測試

下載 .dmg 後拖入 Applications，啟動 Docker Desktop，等狀態列 🐳 圖示變成穩定狀態，在 Terminal 測試：

docker --version
docker run hello-world
# 出現 "Hello from Docker!" 代表成功
🐧 Linux（Ubuntu）步驟 1：移除舊版本

Ubuntu 預設 apt 裡的 docker.io 版本較舊，建議從官方 repository 安裝最新版：

# 移除舊版（若有）
sudo apt remove docker docker-engine docker.io containerd runc

# 更新套件清單
sudo apt update
sudo apt install ca-certificates curl gnupg
🐧 Linux（Ubuntu）步驟 2：加入官方 Repository 並安裝
# 加入 Docker 官方 GPG key
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# 加入 repository
echo "deb [arch=$(dpkg --print-architecture) \
  signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list

# 安裝 Docker Engine
sudo apt update
sudo apt install docker-ce docker-ce-cli containerd.io docker-compose-plugin
🐧 Linux（Ubuntu）步驟 3：設定非 root 使用者

預設需要 sudo 才能用 docker，建議把自己加入 docker 群組，就不用每次打 sudo：

sudo usermod -aG docker $USER
# 登出再重新登入後生效

# 測試
docker run hello-world   # 不用 sudo
💡 本章提示卡
🟢 實用技巧
macOS 使用者可以在 Docker Desktop → Settings → General 裡開啟「Start Docker Desktop when you log in」，這樣開機就自動啟動，不用每次手動開。
🟡 注意事項
Apple Silicon Mac pull Image 時如果看到「WARNING: The requested image's platform (linux/amd64) does not match...」，加上 --platform linux/amd64 可用 Rosetta 2 模擬，但效能較差。
🔴 常見錯誤
Linux 裝完忘記把使用者加進 docker 群組，每次都要打 sudo，更慘的是 docker-compose 或腳本裡沒有 sudo 就會失敗。
🔵 補充說明
Linux 不需要裝 Docker Desktop，直接裝 Docker Engine 就好。Docker Desktop on Linux 是另一個付費產品，一般開發者用不到。
第 5 章跨平台開發專案最佳實務（路徑與專案結構）
📁 步驟 1：各平台建議的專案存放位置

專案放哪裡直接影響 Docker Volume 掛載的效能與相容性，請照以下建議：

🪟 Windows + WSL2
✅ 建議：/home/yourname/projects/
放在 Ubuntu 的本機檔案系統，效能最好。

❌ 不建議：/mnt/c/Users/...
跨 WSL2 和 Windows 的檔案系統，I/O 效能差 10 倍以上。
🍎 macOS
✅ 建議：~/Projects/ 或 ~/Developer/
直接在 macOS 本機檔案系統，IDE 直接開啟。

避免把專案放在雲端同步資料夾（如 iCloud Drive），可能影響 Docker 掛載。
🐧 Linux（Ubuntu）
✅ 建議：/home/yourname/projects/
標準做法，效能最佳，直接在本機操作。

避免放在 /tmp（重開機會清空）或 /root/（權限問題）。
🗂️ 步驟 2：建議的專案目錄結構

建議在每個 Docker 專案裡維持以下結構，讓 Compose 和 Volume 掛載路徑一致：

my-project/
├── docker-compose.yml    # 服務定義
├── .env                  # 環境變數（不要 commit）
├── .gitignore
├── app/                  # 你的應用程式碼
│   ├── Dockerfile
│   └── src/
└── data/                 # 本機持久化資料（可加進 .gitignore）
💻 步驟 3：用 VS Code 開啟 WSL2 專案（Windows）

Windows 使用者安裝 VS Code 的「WSL」擴充套件後，在 Ubuntu Terminal 裡執行 code . 就能直接用 VS Code 開啟，左下角會顯示「WSL: Ubuntu」。

# 在 WSL2 Ubuntu Terminal 裡
cd /home/yourname/projects/my-project
code .   # 自動在 VS Code 以 WSL 模式開啟
💡 本章提示卡
🟢 實用技巧
Windows 使用者可以在 Windows 檔案總管輸入 \\wsl$\Ubuntu\home\yourname\projects 直接存取 WSL2 的檔案，方便複製貼上，但日常開發還是在 WSL2 Terminal 裡操作。
🟡 注意事項
不要把 .env 檔案 commit 進 Git！裡面通常有資料庫密碼、API Key，記得在 .gitignore 加上 .env。
🔴 常見錯誤
Windows 使用者把專案放在 /mnt/c/Users/ 然後掛載進 Docker，導致檔案同步極慢、熱重載失效。專案一定要放在 Ubuntu 本機路徑。
🔵 補充說明
macOS 上 Docker Desktop 有內建檔案共享機制（VirtioFS），把 ~/ 下的路徑掛載進容器效能已大幅改善，不需要額外設定。
第 6 章第一個 Container：從 Hello World 開始
🚀 步驟 1：跑第一個 Container

三個平台的指令完全一樣，在各自的 Terminal 執行：

docker run hello-world
# 成功後會看到 "Hello from Docker!" 的訊息
🔍 步驟 2：理解 docker run 做了什麼

docker run 背後依序做了四件事：在本機找 Image → 找不到就從 Hub pull → 建立 Container → 執行。

docker run nginx
# 1. 本機找 nginx Image → 沒有
# 2. 從 Docker Hub pull nginx:latest
# 3. 建立 Container
# 4. 啟動 nginx（前景執行，Ctrl+C 停止）

# 加上 -d 讓它在背景執行
docker run -d -p 8080:80 nginx
# 然後開瀏覽器看 http://localhost:8080
📋 步驟 3：常用基本指令
docker ps           # 列出執行中的 Container
docker ps -a        # 列出所有 Container（含已停止）
docker stop <id>   # 停止 Container
docker rm <id>     # 刪除 Container（需先停止）
docker images       # 列出本機所有 Image
docker rmi <image> # 刪除 Image
🎛️ 步驟 4：常用 docker run 參數
docker run -d nginx              # -d 背景執行
docker run -p 8080:80 nginx      # -p 主機埠:容器埠
docker run -v /data:/app/data nginx  # -v 掛載 Volume
docker run --name my-nginx nginx # --name 指定名稱
docker run -e KEY=VALUE nginx    # -e 設環境變數
docker run --rm nginx            # --rm 結束後自動刪 Container
🖥️ 步驟 5：進入 Container 內部

想進去 Container 裡看看、執行指令，用 exec -it：

docker exec -it my-nginx bash   # 進入 Container 的 bash
docker exec -it my-nginx sh     # 若沒有 bash 用 sh
# 在裡面可以執行任何 Linux 指令
# 輸入 exit 或 Ctrl+D 離開
💡 本章提示卡
🟢 實用技巧
用 docker run --rm -it ubuntu bash 可以快速開一個乾淨的 Ubuntu 環境測試指令，結束後自動刪掉，不留垃圾。
🟡 注意事項
docker stop 是優雅停止（等 Container 處理完），docker kill 是強制停止。正常情況用 stop，緊急才用 kill。
🔴 常見錯誤
看到 "port is already allocated" 代表本機那個 port 已經被佔用，換一個主機 port（如 -p 8081:80）或先停掉佔用該 port 的程序。
🔵 補充說明
-p 8080:80 的格式是「主機 port：容器 port」，左邊是你在瀏覽器輸入的，右邊是容器裡的服務 port，不要搞反。
第 7 章Image 管理與 Docker Hub
🔍 步驟 1：搜尋與下載 Image
docker search nginx             # 搜尋 Docker Hub 上的 Image
docker pull nginx               # 下載最新版（latest）
docker pull nginx:1.25          # 下載特定版本
docker pull mysql:8.0
docker images                   # 查看本機所有 Image
🏗️ 步驟 2：用 Dockerfile 建立自己的 Image

Dockerfile 是建立 Image 的指令檔，每一行指令都是一層（layer）：

# Dockerfile 範例（Node.js App）
FROM node:20-alpine          # 基底 Image
WORKDIR /app                 # 設定工作目錄
COPY package*.json ./        # 複製 package.json
RUN npm install              # 安裝依賴
COPY . .                     # 複製所有檔案
EXPOSE 3000                  # 聲明容器用的 port
CMD ["node", "index.js"]     # 啟動指令
🔨 步驟 3：build 與 tag
# 在 Dockerfile 所在目錄執行
docker build -t myapp:1.0 .
# -t 指定名稱:版本，. 表示目前目錄

docker tag myapp:1.0 myapp:latest  # 加上 latest tag
docker images                       # 確認建立成功
⬆️ 步驟 4：推送 Image 到 Docker Hub
docker login                        # 登入 Docker Hub
docker tag myapp:1.0 yourusername/myapp:1.0
docker push yourusername/myapp:1.0  # 推上去
🧹 步驟 5：清理本機 Image
docker rmi nginx:1.25       # 刪除特定 Image
docker image prune          # 刪除沒有 tag 的 Image（dangling）
⚠️ 警告：docker image prune -a
  • docker image prune -a 會刪除所有沒有被任何 Container 使用的 Image，包括你下載但沒跑的 Image。
  • 執行前確認哪些 Image 你還需要，刪掉後需要重新 pull 才能用。
💡 本章提示卡
🟢 實用技巧
選 Base Image 時優先選 alpine 版本（如 node:20-alpine），檔案比完整版小 10 倍以上，build 快、push 快、pull 也快。
🟡 注意事項
Dockerfile 的每一行都是一個 layer，layer 越少越好。把多個 RUN 指令用 && 串在一起可以減少 layer 數量，讓 Image 更小。
🔴 常見錯誤
把 .env、node_modules、.git 等不必要的檔案 COPY 進 Image。建立 .dockerignore 排除它們（格式跟 .gitignore 一樣）。
🔵 補充說明
Multi-stage build 可以在 build 階段用完整環境編譯，最後只複製執行檔到輕量 Image，讓 Production Image 極小化。適合編譯型語言（Go、Java、TypeScript）。
第 8 章Volume 與資料持久化（含平台差異）
💾 步驟 1：為什麼需要 Volume

Container 刪掉之後，裡面的所有資料也會消失。Volume 讓資料「活在 Container 之外」，Container 重建後資料還在。

# 沒有 Volume：Container 刪掉，資料消失
docker rm mysql-container   # DB 資料全沒了！

# 有 Volume：資料存在 Volume，Container 可以重建
docker volume create mysql-data
docker run -v mysql-data:/var/lib/mysql mysql:8.0
📂 步驟 2：兩種掛載方式

Named Volume：由 Docker 管理，推薦用於資料庫等持久化資料。Bind Mount：直接掛載主機目錄，推薦用於開發時的程式碼同步。

# Named Volume（Docker 管理存放位置）
docker run -v mysql-data:/var/lib/mysql mysql:8.0

# Bind Mount（掛載主機目錄）
docker run -v /home/user/app:/app node:20
🗺️ 步驟 3：各平台 Bind Mount 路徑範例
🪟 Windows + WSL2
docker run -v \
  /home/user/projects/app:/app \
  node:20
# 專案放在 Ubuntu 本機路徑
# 不要用 /mnt/c/...
🍎 macOS
docker run -v \
  /Users/user/Projects/app:/app \
  node:20
# 或用 ~ 縮寫
docker run -v ~/Projects/app:/app node:20
🐧 Linux
docker run -v \
  /home/user/projects/app:/app \
  node:20
# 跟 macOS 類似，用 ~ 也可以
🛠️ 步驟 4：Volume 管理指令
docker volume ls                    # 列出所有 Volume
docker volume inspect mysql-data    # 查看 Volume 詳細資訊
docker volume rm mysql-data         # 刪除 Volume
⚠️ 警告：docker volume rm 與 docker volume prune
  • docker volume rm <name>：永久刪除指定 Volume 的所有資料，無法復原。
  • docker volume prune：刪除所有沒有被任何 Container 使用的 Volume，包括你以為還在用的 DB 資料。
  • 執行前務必確認，或先備份資料。資料庫資料刪掉就真的沒了。
💡 本章提示卡
🟢 實用技巧
開發時用 Bind Mount 掛載程式碼目錄，這樣你在 IDE 裡修改的程式碼，Container 裡馬上看得到，不用重 build Image。
🟡 注意事項
Windows WSL2 用戶：Bind Mount 的路徑要用 Ubuntu 的路徑（/home/...），不要用 Windows 路徑（C:\...）或 WSL 的 /mnt/c/...，效能差很多。
🔴 常見錯誤
刪掉 Container 以為資料還在，結果沒設 Volume 導致資料全沒了。DB Container 一定要設 Volume！
🔵 補充說明
Named Volume 的資料實際存在哪裡？Linux 在 /var/lib/docker/volumes/，macOS/Windows 在 Docker Desktop 的虛擬磁碟裡，不是 Host 的本機路徑。
第 9 章Container Network 與 Port 映射
🌐 步驟 1：Container 的網路基本概念

每個 Container 都有自己的網路空間，預設加入 bridge network。Container 之間可以透過容器名稱互相連線，不需要透過主機。

docker network ls               # 列出所有 network
# NETWORK ID   NAME      DRIVER
# abc123       bridge    bridge   ← 預設
# def456       host      host
# ghi789       none      null
🔌 步驟 2：Port 映射（-p）

Container 裡的服務預設外界無法存取，要用 -p 把容器 port 映射到主機 port：

# 格式：-p 主機port:容器port
docker run -d -p 8080:80 nginx
# 瀏覽器開 http://localhost:8080

docker run -d -p 3306:3306 mysql:8.0
# 本機 MySQL client 連 localhost:3306

# 多個 port 映射
docker run -d -p 80:80 -p 443:443 nginx
🔗 步驟 3：Container 之間的通訊（自訂 network）

同一個 network 裡的 Container 可以用容器名稱互連，不需要 IP：

# 建立自訂 network
docker network create my-net

# 把兩個 Container 加入同一個 network
docker run -d --name app --network my-net myapp
docker run -d --name db --network my-net mysql:8.0

# app Container 裡可以用 "db" 當主機名連到 MySQL
# mysql -h db -u root -p
🔍 步驟 4：查看 Container 網路資訊
docker inspect my-container     # 查看完整資訊（含 IP）
docker network inspect my-net   # 查看 network 內的 Container
💡 本章提示卡
🟢 實用技巧
用 Docker Compose 時不需要手動建 network，Compose 會自動建一個 network 讓所有服務互通，直接用服務名稱（如 db）當主機名連線就好。
🟡 注意事項
-p 只把 port 開放給主機，不代表開放給外部網路。如果要讓其他機器也能連，加 -p 0.0.0.0:8080:80 或在防火牆開放該 port。
🔴 常見錯誤
兩個 Container 都用預設 bridge network，發現用容器名稱連不到對方——預設 bridge network 不支援 DNS 解析，要用自訂 network 才行。
🔵 補充說明
--network host 讓 Container 直接用主機的 network，不需要 -p 映射，但這樣 Container 的隔離性降低，僅在 Linux 上有效，macOS/Windows 不支援。
第 10 章Docker Compose：多服務協作實戰
📄 步驟 1：認識 docker-compose.yml

Compose 讓你用一個 YAML 檔定義所有服務，一個指令啟動整個開發環境：

# docker-compose.yml 基本結構
services:
  app:                          # 服務名稱
    build: .                    # 用當前 Dockerfile build
    ports:
      - "3000:3000"
    depends_on:
      - db
    environment:
      DB_HOST: db               # 用服務名稱連 DB
      DB_PORT: 3306

  db:
    image: mysql:8.0            # 直接用現成 Image
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: myapp
    volumes:
      - mysql-data:/var/lib/mysql

volumes:
  mysql-data:                   # 宣告 Named Volume
▶️ 步驟 2：常用 Compose 指令
docker compose up -d            # 背景啟動所有服務
docker compose up --build -d    # 重新 build 後啟動
docker compose down             # 停止並移除 Container
docker compose down -v          # 停止並刪除 Volume（資料會消失！）
docker compose logs -f          # 即時查看所有服務 log
docker compose logs -f app      # 只看 app 服務的 log
docker compose ps               # 查看服務狀態
docker compose exec app bash    # 進入 app Container
🌍 步驟 3：用 .env 管理環境變數

把敏感資訊放在 .env，Compose 會自動讀取，不用寫死在 yml 裡：

# .env 檔案
DB_PASSWORD=mysecretpassword
DB_NAME=myapp
APP_PORT=3000

# docker-compose.yml 裡這樣用
environment:
  MYSQL_ROOT_PASSWORD: ${DB_PASSWORD}
  MYSQL_DATABASE: ${DB_NAME}
ports:
  - "${APP_PORT}:3000"
🔄 步驟 4：開發環境熱重載設定

開發時 Bind Mount 程式碼進 Container，加上 nodemon 等工具，改了程式碼自動重啟：

services:
  app:
    build: .
    volumes:
      - ./app:/app              # Bind Mount 程式碼
      - /app/node_modules       # 保留 Container 裡的 node_modules
    command: npm run dev        # 用 nodemon 等開發模式啟動
⚠️ 警告：docker compose down -v
  • docker compose down -v 加上 -v 會刪除所有 Compose 定義的 Volume，包括資料庫的資料。
  • 只有在你確定要「完全重置」開發環境時才加 -v，日常重啟用 docker compose down 就好。
💡 本章提示卡
🟢 實用技巧
docker compose up --build 在每次啟動時重新 build Image，適合程式碼有更動時用。日常重啟用 docker compose up -d 比較快。
🟡 注意事項
depends_on 只保證 Container 的啟動順序，不保證服務已經 ready（DB 可能還在初始化）。App 連 DB 時要加重試邏輯。
🔴 常見錯誤
在 docker-compose.yml 裡 hardcode 密碼，然後 commit 進 Git。一定要用 .env 檔管理敏感資訊，並把 .env 加進 .gitignore。
🔵 補充說明
新版 Docker 已把 Compose 整合進 CLI（docker compose），舊版是獨立的 docker-compose（有連字號）。建議用新版，功能更完整。
第 11 章開發流程整合與除錯技巧
📋 步驟 1：查看 Container Log
docker logs my-container        # 查看 log
docker logs -f my-container     # 即時追蹤 log（like tail -f）
docker logs --tail 50 my-container  # 只看最後 50 行
docker compose logs -f app      # Compose 服務 log
🔍 步驟 2：進入 Container 除錯
docker exec -it my-container bash   # 進入 bash
docker exec -it my-container sh     # 沒有 bash 用 sh
# 在裡面可以：
# - 看檔案：ls -la /app
# - 查環境變數：env
# - 測試網路：curl http://db:3306
# - 看 process：ps aux
📊 步驟 3：查看資源使用狀況
docker stats                    # 即時 CPU / Memory 使用量
docker inspect my-container     # 完整 Container 資訊（JSON）
docker top my-container         # 查看 Container 內的 process
🧹 步驟 4：清理開發環境
docker system df                # 查看 Docker 佔用的磁碟空間
docker system prune             # 清除停止的 Container、沒用的 Image 和 network
⚠️ 警告：docker system prune -a --volumes
  • 這個指令會清除「所有停止的 Container、所有沒有 tag 的 Image、所有 network、以及所有未使用的 Volume」。
  • 開發環境用這個可能會刪掉你以為還在的資料庫 Volume，導致資料消失。
  • 建議只用 docker system prune（不加 --volumes），或明確指定要清理的項目。
💡 本章提示卡
🟢 實用技巧
VS Code 安裝「Docker」擴充套件後，可以在 GUI 裡直接看到所有 Container / Image / Volume，右鍵就能 exec、看 log，比打指令方便很多。
🟡 注意事項
Container 裡修改的檔案，在 Container 刪掉後就消失了（除非有 Bind Mount 或 Volume）。不要在 Container 裡直接修改設定，要在 Dockerfile 或 Compose 裡設定。
🔴 常見錯誤
Container 一直重啟（CrashLoopBackOff），但用 docker logs 卻看不到有用的資訊——加上 --tail 100 或 -f 追蹤，也可以用 docker events 看 Docker 層級的事件。
🔵 補充說明
Healthcheck 可以讓 Docker 知道你的服務是否真的 ready，在 Dockerfile 或 Compose 裡加 healthcheck 設定，docker ps 就會顯示健康狀態。
第 12 章常見錯誤排查與回溯救援
❌ 情境 1：Cannot connect to the Docker daemon
# 錯誤訊息
Cannot connect to the Docker daemon at unix:///var/run/docker.sock

# 排查步驟
# Windows / macOS：確認 Docker Desktop 有開啟並完全啟動
# Linux：
sudo systemctl status docker    # 確認 Docker daemon 是否在跑
sudo systemctl start docker     # 若沒跑就啟動
❌ 情境 2：Port is already allocated
# 錯誤訊息
Error: bind: address already in use

# 排查：找出誰佔用了 port
# Linux / macOS
lsof -i :8080
sudo kill -9 <PID>

# Windows（PowerShell）
netstat -ano | findstr :8080
taskkill /PID <PID> /F

# 或直接換一個主機 port
docker run -p 8081:80 nginx
❌ 情境 3：Volume 資料不見了
# 確認 Volume 是否存在
docker volume ls

# 若找到 Volume 但 Container 找不到資料
# 確認 Compose 的 volume 名稱是否正確
docker volume inspect <volume-name>
# 查看 Mountpoint 欄位確認資料位置
❌ 情境 4：Image build 失敗
# 看完整 build log 找錯誤
docker build -t myapp . --progress=plain

# 常見原因：
# 1. COPY 的檔案不存在 → 確認路徑和 .dockerignore
# 2. RUN npm install 失敗 → 確認 package.json 有複製進去
# 3. 網路問題 → 換 npm registry 或用 --network host
❌ 情境 5：WSL2 特有問題（Windows）
# Docker Desktop 沒整合 WSL2
# 去 Settings → Resources → WSL Integration → 勾選 Ubuntu

# WSL2 記憶體佔用太高
# 在 C:\Users\yourname\.wslconfig 加入：
[wsl2]
memory=4GB
swap=2GB

# 套用設定
wsl --shutdown
💡 本章提示卡
🟢 實用技巧
遇到問題先執行 docker logs <container> 和 docker inspect <container>，90% 的問題在這兩個指令的輸出裡找得到線索。
🟡 注意事項
Apple Silicon Mac 遇到 exec format error，通常是跑了 x86 Image 但沒加 --platform linux/amd64，或是 build 時沒指定 platform。
🔴 常見錯誤
Compose 改了設定（如 port、volume）但 docker compose up -d 沒生效——需要先 docker compose down 再重新 up，才會套用新設定。
🔵 補充說明
docker events 可以即時監聽 Docker daemon 的所有事件（Container 啟動、停止、die 等），適合排查 Container 為什麼一直重啟。
第 13 章常見情境速查與團隊最佳實務
⚡ 情境速查：日常開發常用
# 啟動開發環境
docker compose up -d

# 查看狀態
docker compose ps
docker compose logs -f

# 重新 build 某個服務
docker compose up --build app -d

# 進入 Container
docker compose exec app bash

# 關閉開發環境（保留資料）
docker compose down

# 完全重置（刪除資料，慎用）
docker compose down -v
🌿 步驟 2：建議的分支與環境對應
# 建議在專案裡維護兩個 Compose 設定
docker-compose.yml          # 開發環境（含 hot reload、debug port）
docker-compose.prod.yml     # 正式環境（無 bind mount、資源限制）

# 指定設定檔啟動
docker compose -f docker-compose.prod.yml up -d
📋 步驟 3：團隊規範建議
# 每個專案必備檔案
.env.example          # 環境變數範本（commit 進 git）
.env                  # 實際環境變數（加進 .gitignore）
.dockerignore         # 排除不需要的檔案
docker-compose.yml    # 開發環境定義
README.md             # 說明如何啟動開發環境
🔒 步驟 4：安全性基本注意
# 不要用 root 跑 App
USER node              # 在 Dockerfile 裡切換使用者

# 不要把敏感資訊寫在 Dockerfile 或 Image 裡
# 用 .env 或 Docker Secrets 管理

# 定期更新 Base Image
docker pull node:20-alpine   # 重新 pull 最新版
docker compose up --build    # 重新 build
💡 本章提示卡
🟢 實用技巧
在 README.md 裡加一段「快速開始」，讓新成員只要執行 cp .env.example .env 然後 docker compose up -d 就能跑起整個開發環境，不需要看落落長的設定文件。
🟡 注意事項
不同平台的團隊成員（Windows WSL2 / macOS / Linux）在 Volume 路徑和換行符上可能有差異，建議在 .gitattributes 統一換行符，並在文件裡標注各平台的注意事項。
🔴 常見錯誤
直接把 Production 的 docker-compose.yml 拿來開發用，沒有 Bind Mount 導致每次改程式碼都要重 build，浪費大量時間。開發和正式環境要分開設定。
🔵 補充說明
CI/CD 環境（GitHub Actions / GitLab CI）也可以用 Docker 跑測試，確保「測試環境 = 開發環境 = 正式環境」的一致性，這是導入 Docker 最大的價值之一。
